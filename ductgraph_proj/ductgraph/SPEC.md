1) 2) maxload 初期推定：枝の合計損失最大

これは良い方針です。“合計損失”の定義だけを設計上固定します。

合計損失（初期推定用）の定義

初期推定では、まだ流量分配が無いので 設計風量で評価します。

各端末枝 i について、設計風量 Q_i を仮定（CAVなら setpoint、hood付きも同様）

そのときの枝の圧損

直管（Darcy–Weisbach）

局部（K）

フード（指定 dp）

Tee（分岐側/直進側 K：設計流量比で評価） を全部足して Δp_i(Q_i) を求める


Δp_i(Q_i) が最大の端末枝を maxload 初期値にする


> 実務的に「最も苦しい枝＝最大圧損枝」を maxload 起点にするので、収束も早いはずです。

3) 4) Tee は「入口2・出口1」のみ（複数分岐は想定しない）

了解です。したがってモデルは **二分岐（binary）**のみ。

あなたの構造は：

T1：幹線 350φ → a(250φ) 分岐 + b 幹線(350φ) 直進

T2：b 幹線 350φ → c(250φ) 分岐 + d(250φ) 分岐ではなく、ここは「入口2出口1」制約と整合しないように見えますが、図を読むと実際は


> b幹線 → T2 →（c と d に分かれる）



なので、T2 は 入口1・出口2（分岐）ですよね。

ここ、言葉の整理が必要です。設備の Tee は通常「入口1 → 出口2（分岐）」ですが、あなたが言っている「入口2・出口1」は 合流側の見方です。

結論としては、計算モデル上は同じで、向きをどちらに取るかだけです。

**ductgraph は “有向 Edge”**なので、Tee は「合流」でも「分岐」でも書けます

ただし K テーブルの参照が「branch/straight」である以上、

ファンから下流方向を基準に「分岐」として統一するのが実装が単純です



以後、Tee は 分岐（1→2）として表現し、K の branch/straight を適用します。

6) 「Edge として入れる」vs「Edge の r に加算」—何が違うか（数学モデルは触る？）

結論から言うと：

solver（ニュートン）の数学モデルは触りません

違いは「状態依存（流量比依存）が表現できるか」と「モデルの保守性」です


A. r に加算（現行 solver のまま）

Edge.r に「その枝に属する損失」を全部まとめる

solver は Δp = r_eff * Q|Q| - h を使うだけ


メリット

実装が最小

ニュートンの未知数や Jacobian 変更なし


デメリット

Tee の K が流量比で変わる場合、r は本来 状態依存
→ r を固定値にするなら「設計点での近似」になる
（初期推定や設計一点チェックには十分だが、運転点が動くと誤差が出る）


あなたが「指数モデルでも良い」「抵抗は大きい側に寄せたい」と言っているので、当面は **設計点評価＋安全側（大きめ）**で問題ありません。

B. Tee を別 Edge として入れる

Tee を「局部抵抗エッジ」として分離し、例えば

幹線→tee node→枝 のように追加エッジを挟む形。


メリット

内訳が明確で、デバッグしやすい

将来「状態依存 K」を入れる拡張の足場になる


デメリット

Tee は本来「ノードでの流量配分」に依存するので、 ただエッジ分割しただけでは状態依存は解決しない

ノード増加で収束性や入力が複雑化する


結論

当面は **A（r に加算）**で進めるのが合理的

ただし Tee の K は「設計流量比で評価→安全側に寄せる」ルールを決める

8) ここだけ選んでください（曖昧だと事故る唯一の点）

あなたが「抵抗は大きい側に寄せたい」と言っているので、設計としてどちらかに統一します。

選択肢（どちらも solver は触りません）

(H1) hood を一定圧損（dp一定）として扱う
→ Edge.h（もしくは別枠）で表現。Q 依存しない。

(H2) hood を等価抵抗 r に換算して扱う（設計点で）
→ r_hood = dp_hood / (Q_design^2)（m³/s系）として r に加算。
ただし Q が変わると本来ズレる。


現場の「フードはある程度 dp が決まる（装置側で支配される）」なら H1 が物理的に自然です。
一方で「全部 r にまとめて安全側に寄せたい」なら H2 が実装が単純です。

あなたの方針（安全側＝抵抗大きめ）からすると、当面は H2 が一貫します。
